  - 参数错误：spacing 或仓位上限设置不当

  - 挂单总能及时成交或撤销

- **偏差是否方向性有利于策略**：  
  否。偏差倾向于**放大风险**，而非低估风险。

#### Ⅵ. Required Execution Controls（必须的执行控制）

- **必须上线的执行保护**：
  1. 挂单成交概率监控（实时计算订单深度与队列位置）
  2. 撤单失败检测与重试机制
  3. 执行成本阈值（如果滑点超过 X%，禁止新增仓位）
  4. 部分成交处理逻辑

- **撤单/成交失败时的 fallback 机制**：
  1. 撤单失败 → 立即转为 taker 主动平仓
  2. 成交失败 → 降低仓位上限，禁止新增风险
  3. 连续失败 → 进入 EMERGENCY STOP

- **对策略逻辑的执行层改造建议**：
  1. 在 DEFENSIVE 状态下，不仅冻结新增 buy，还应主动撤销最外层 buy
  2. 在 DAMAGE_CONTROL 状态下，应使用 taker 主动减仓，而非等待挂单成交
  3. 增加“订单健康度”监控，实时评估执行质量

## B. 风险结构与错配总结

### 主要风险结构

1. **Negative Convexity（负凸性）**：
   - 策略本质为 short gamma，损失在极端行情下非线性放大
   - 风险结构：损失 = f(价格偏离)²，而非线性关系

2. **Path-Dependent Loss（路径依赖损失）**：
   - inventory 累积速度取决于价格路径，而非终点
   - 风险结构：损失 = f(路径积分)，而非终点价格

3. **Liquidity-Dependent Exit（流动性依赖退出）**：
   - 减仓能力依赖市场流动性
   - 风险结构：在需要退出时，流动性可能收缩

### 主要错配情景

1. **负凸性 × 趋势行情**：
   - 市场进入持续方向性运动，区间假设失效
   - 风险放大：inventory 持续累积，损失非线性增长

2. **路径依赖 × 波动跳变**：
   - 波动率突然放大，但价格仍在区间内
   - 风险放大：执行成本非线性上升，减仓困难

3. **流动性依赖 × 流动性收缩**：
   - 在需要减仓时，流动性突然收缩
   - 风险放大：无法有效执行减仓，风险暴露持续存在

### 框架的缓解措施

1. **多层次停机机制**：
   - DEFENSIVE → DAMAGE_CONTROL → EMERGENCY STOP
   - 在不同风险级别下，采取不同的控制措施

2. **仓位优先原则**：
   - inventory 是第一风险变量
   - 在边界位置，禁止新增仓位

3. **边缘递减机制**：
   - 在接近区间边缘时，单层下单 size 递减
   - 在边缘缓冲区，禁止新增仓位，只允许减仓
